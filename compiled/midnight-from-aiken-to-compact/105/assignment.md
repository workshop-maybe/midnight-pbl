# Module Assignment

## Your role

You're not proving you learned this — you're helping build it. This course is a community resource, and every module is a draft that gets better through the people who use it. Your credential means you engaged with the material deeply enough to improve it.

## What to do

Work through the three lessons in this module, then respond to the following.

### 1. What real-world use case is missing?

This module teaches three credential patterns: signature-based verification, MerkleTree membership, and selective disclosure. The lessons use examples (professional certifications, organization membership, age verification). What use case from your own domain would make the patterns more tangible? Describe it in enough detail that it could become a lesson example — what's being proven, what stays private, and which pattern applies.

### 2. Where do the patterns need more context?

The three credential patterns are presented separately across three lessons. Where would a developer struggle to combine them in practice? Identify a gap: maybe the transition from single-credential verification to multi-credential membership isn't well motivated. Maybe the selective disclosure levels (boolean, derived value, partial attribute) need a decision framework for when to use each one.

### 3. Propose a revision

Choose one of:

- **A new example** for an existing pattern that's more compelling or clearer than the current one. Write it up with enough detail to slot into the lesson.
- **A bridge section** that connects two patterns the lessons treat separately. When would you combine signature verification with MerkleTree membership? Write the explanation.
- **A rewrite** of the weakest section in this module — the part where the conceptual explanation doesn't match what a developer would actually need to implement it.

## What makes a strong submission

- **Ground it in reality** — credential systems are only interesting if the use case is real. Abstract pattern descriptions are the failure mode of this module; concrete applications are the fix.
- **Identify the missing "why"** — the lessons explain *how* each pattern works. Where do they fail to explain *when* to choose one over another?
- **Think like a builder** — if you were implementing a credential system tomorrow, what would this module need to say that it currently doesn't?
